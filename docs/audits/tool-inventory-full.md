## Architecture

- Live runtime registration starts in `lib/tools.js`. The static `tools` array is the source of truth for public tool slugs.
- Next.js routing is explicit: each live tool has a file at `app/tools/<slug>/page.js` and is therefore reachable at `/tools/<slug>`.
- Each live page imports one wrapper component from `components/tools/widgets/<Name>Tool.js`.
- Each wrapper is a thin shell around `components/tools/widgets/UniversalToolWidget.js`, which branches on the hard-coded slug and runs the tool logic.
- Tool prose, FAQs, schema data, related links, and metadata all flow from helpers in `lib/tools.js`, especially `getToolContent()`, `buildToolMetadata()`, and `buildToolSchemas()`.
- Staged tools differ in two important ways: they live outside `app/tools`, and their page templates assume future integration into the live widget/registry system.
- Staged pages currently call live helpers such as `getToolContent(slug)`, but their slugs are not present in `lib/tools.js`, so they are not runtime-safe today.
- Staged widgets are mostly self-contained implementations stored inside their own batch folders, not in the live widget directory.
