## 6. Registration System

- Source of truth for live tool registration is the static `tools` array in `lib/tools.js`.
- The tool directory page at `app/tools/page.js` imports that registry directly and renders search/category exploration from it.
- Category pages at `app/tools/category/[category]/page.js` derive category routes from the same registry via `getAllCategorySlugs()` and `getToolsByCategory()`.
- Each live tool has an explicit route file at `app/tools/<slug>/page.js`.
- Each route explicitly imports one wrapper component from `components/tools/widgets/<Name>Tool.js`.
- Each wrapper component delegates to `components/tools/widgets/UniversalToolWidget.js` with a hard-coded `slug` prop.
- SEO metadata, FAQs, related tools, schemas, and tool prose are generated from helpers in `lib/tools.js` using `buildToolMetadata()`, `getToolContent()`, and `buildToolSchemas()`.
- No dynamic imports were detected in the live tool registration flow.
- No config-driven runtime loader was detected for live tools.
- `generated-bulk-tools/` is not referenced by runtime code under `app/`, `components/`, `lib/`, or `scripts/`; its README explicitly marks it as staging-only.

### Circular dependencies

- No circular dependency mechanism was detected in the live registration path reviewed for this audit.
- The core flow is one-way: `lib/tools.js` -> route pages -> wrapper widgets -> `UniversalToolWidget.js`, with helper consumers in layout/info/schema components.

### Empty tool folders

- None detected in scanned tool directories after excluding the dynamic route segment `app/tools/category/[category]` from false-positive folder checks

### Additional observations

- `generated-bulk-tools/` is staging only and is not referenced by the runtime. Its 56 tools are currently unreachable from Next.js routing.
- Bulk page files import widgets from `@/components/tools/widgets/...`, but the staged widget implementations live inside the batch folders. The staged pages are templates for merge work, not runnable in place.
- The live system is highly synchronized by convention: adding a tool requires a registry entry in `lib/tools.js`, a route under `app/tools/<slug>/page.js`, and a wrapper component under `components/tools/widgets/<Name>Tool.js`.
- All 60 live widget wrapper files delegate into one large `UniversalToolWidget.js`, which centralizes behavior but increases coupling and raises regression risk when editing one tool behavior.
- `app/tools/category/[category]/page.js` contains `console.log` statements in server-side route code, which may add noise in production or server logs.

## 8. Recommendations

- Decide whether `generated-bulk-tools/` should remain a staging archive or become an integration pipeline. Right now it is a disconnected inventory, not an executable extension of the app.
- If you want to merge bulk tools, introduce a single manifest-driven generator so one source creates the registry entry, route file, wrapper file, metadata, and optional widget implementation together.
- Consider replacing 60 thin wrapper files plus 60 route files with a manifest-based dynamic route and a smaller component registry. That would reduce synchronization drift.
- Split `UniversalToolWidget.js` into focused modules by capability family such as counters, converters, analyzers, audio, and privacy tools.
- Remove server-side `console.log` calls from `app/tools/category/[category]/page.js` after debugging is complete.
- Add an audit or CI check that verifies every registry slug has exactly one route, exactly one widget import, and zero duplicates.
